Basics

Getting started

Back to contents
Navigation, Favorites, and finding tools

The persistent left pane is the primary navigation. Sections collapse independently, the current section opens automatically, and the pane scrolls separately from the page. Help and the installed version remain at the bottom.

  • Favorites contains personal shortcuts. Hover or focus a tool and use its star to add/remove it.
  • Fortinet Tools links to FortiGate and FortiAuthenticator profiles/workflows.
  • Network Tools contains vendor-neutral diagnostics.
  • Automation contains conditions, actions, and automations.
  • Administration contains Settings and areas allowed by your access profiles.

On narrow screens, use the hamburger button to open or close navigation.

First launch and service startup

There is no default login. A fresh installation generates a self-signed HTTPS certificate, prints its HTTPS URL, and the first visit creates the initial administrator. Review the browser's certificate warning before continuing. Rerunning the installer preserves an existing installation's protocol choice.

./install.sh
./twn start
./twn status

The service normally uses port 5050. Override one launch with TWN_TOOLKIT_PORT=8000 ./twn start.

Shared interface conventions
  • Expandable cards contain saved definitions, editors, or result history.
  • New opens a form and changes to Cancel while open.
  • Secret fields are write-only and are not rendered back into forms.
  • Test/preview controls are read-only unless the page explicitly says changes will be applied.
  • Loading cards indicate server-side work. Automation workers continue without an open browser.

Dashboard and metrics

Metrics, time ranges, and scoreboard

Raw counters measure work performed: probes, replies, queries, frames, API calls, and completed tests. Activity score counts deliberate user executions; it does not imply remote success.

Use last hour, 24 hours, 7 days, 30 days, lifetime, or a custom local range. The range applies to widgets, rankings, and recent activity. Expand a scoreboard user to see all non-zero metrics.

Editing and resetting

Administrators can show, hide, and drag widgets into a global order. Hidden widgets move below the divider. Cancel restores the previous layout; Save persists it for all users. The layout can be backed up, but counters/history cannot.

Resetting a metric clears it globally. Clearing a user score resets only that user’s activity score and preserves their raw operational counters.

Administration

Accounts and access

Back to contents
Users and custom access profiles

Administrators create custom profiles and assign one or more to each user. Effective access is the union of assigned profiles. Unauthorized tools disappear from navigation and remain blocked by the server if requested directly.

The built-in Administrator profile is protected. Assign it only to people who should manage users, profiles, backups, server access, and high-impact tools.

Sessions, password policy, and recovery

Settings controls minimum password length, optional complexity, and idle timeout. Never disables idle expiration. If all administrators are locked out:

./twn stop
./twn adminreset
./twn start

This resets users only; saved device profiles and API keys remain.

Saved data

Profiles, secrets, and backups

Back to contents
Saved profiles and credentials

Reusable lists and credentials reduce repeated entry across Fortinet, ping, DNS, RADIUS, SNMP, NTP, traceroute, TCP scanning, and automation tools. API keys, passwords, RADIUS secrets, and SNMP credentials are never rendered back into forms.

Most profile stores are owner-readable local files, not a remote secrets vault. Automation action configuration is encrypted at rest using the installation session secret. Treat the host and instance/ as trusted and sensitive.

Encrypted exports and import behavior

Select the groups to export in Settings → Profile backup and restore. If any selection contains secrets, a backup password is mandatory; otherwise encryption is optional. Anyone with both file and password can restore the secrets.

  • Combine keeps existing entries and adds imported data.
  • Replace replaces selected stored groups.
  • Wrong passwords or damaged encrypted files fail without importing partial plaintext.
  • Imported automations remain paused until reviewed.

Automation history/output, dashboard counters, and recent activity are not included.

Fortinet profiles and workflows

Profiles, permissions, and timeouts

Create and Test FortiGate/FortiAuthenticator profiles before running workflows. Use the least-privileged API administrator that permits the intended action. Test proves connectivity/authentication; HTTP 403 on a later operation usually means the remote profile lacks permission for that endpoint or write.

Unreachable appliances return after bounded timeouts rather than hanging indefinitely.

Exports, rename, reorder, and cleanup
  • Export workflows fetch managed AP, switch, wireless-client, switch-client, and related data into previews/CSV.
  • Rename workflows support CSV and interactive device loading. Review dry runs before applying.
  • Re-order Managed FortiSwitches previews moves and verifies final order.
  • FortiAuthenticator cleanup previews overlap before removing memberships or deleting MAC devices.
Wireless client history

The lookup normalizes MAC addresses, combines live state with local wireless event logs, and focuses on association-request events. Consecutive entries on one AP collapse; a roam creates the next visit. “No logs found” can mean memory logs are unavailable or old events rolled out. FortiAnalyzer is reserved for a future source.

Diagnostics

Network tools

Back to contents
Reachability, path, and service tools
  • Multi-Host Ping: live reachability, latency/loss charts, profiles, locked history, and CSV. Browser history is temporary.
  • Traceroute: UDP/ICMP traces to multiple targets with graphical and text output.
  • Path MTU: finds the largest unfragmented IPv4/IPv6 ICMP packet reaching a target.
  • TCP Port Scanner: checks authorized hosts against ports/ranges.
  • Certificate Inspector: shows the TLS chain and validates hostname, dates, order, and trust.
  • Wi-Fi/LAN Speed Test: browser-to-toolkit performance, not ISP speed.
Protocol and authentication tools
  • Ping: snapshots validated targets at Start; use Update targets to apply edits without clearing existing history.
  • DNS: compares answers/timing across resolvers.
  • SNMP: reusable v2c/v3 credentials, hosts, and OID collections.
  • RADIUS: PAP/CHAP plus decoded attributes; EAP methods require eapol_test.
  • NTP: compares offset, delay, jitter, stratum, identity, and sync health.
  • Syslog: sends RFC 5424 or listens for bounded UDP/TCP messages.
  • Webhook/API Tester: bounded HTTP requests without redirect following.
DHCP, Multi-SSH, Multi-Transfer, and utilities
  • DHCP Discover: inspects Offers without requesting a lease. Port 68/raw interface access may need root/capabilities.
  • Multi-SSH: concurrent prompt-aware commands. Use Friendly Name = host and [timeout=600] command. Results can be exported.
  • Multi-Transfer: fetches the same remote file list concurrently from named hosts over SFTP, SCP, or FTP. Choose a datastore folder for retained files or download a one-shot ZIP whose temporary server files are discarded. The default {timestamp}-{identity}-{filename} pattern can be customized with timestamp, host, friendly-label, identity, filename, stem, and suffix tokens. Duplicate names receive a numeric suffix; the ZIP also contains a transfer report. Unknown SSH host keys remain blocked unless explicitly allowed. FTP is plaintext and should be reserved for legacy targets.
  • Subnet Excluder: subtracts IPv4/IPv6 child networks.
  • What’s My IP?: shows the toolkit-observed client and attempts a browser-side public lookup.
Host-side utilities

Local tools and datastore

Back to contents
Contained file management

The Datastore manages persistent files beneath instance/datastore/. Access is assigned like any other tool through custom profiles. Users can create and browse folders, upload multiple files, download files, rename entries, and delete files or empty folders.

Uploads are atomic and limited to 1 GiB per request. Existing names are never overwritten. The manager rejects traversal outside its root, does not follow symbolic links, and does not recursively delete populated folders.

Switch between list and grid layouts; the browser remembers the preference. Select files or folders to reveal bulk move/delete controls, or drag selected items directly onto a folder. Non-empty folders cannot be deleted accidentally, and folders cannot be moved into themselves. The upload target accepts one or more files through the picker or desktop drag-and-drop. Bulk operations validate every source and destination before changing files.

Managed TFTP service

Local Tools → File Transfers contains managed TFTP, FTP, and authenticated SFTP/SCP listeners; outbound retrieval is available separately under Network Tools → Multi-Host Tools → Multi-Transfer. All listeners are disabled by default and remain contained beneath a selected datastore folder or one runtime-only file.

The SSH transfer listener defaults to 127.0.0.1:2022, stores only a password hash, rejects untrusted CIDRs before authentication, and never provides an interactive shell. SFTP and SCP, downloads, atomic uploads, overwrites, and incoming filename rewriting are independently controlled.

The TFTP root may be the datastore root or any saved subfolder. Alternatively, temporary mode serves one uploaded file that is erased whenever TFTP stops or restarts. Temporary mode is download-only.

The FTP listener has configurable control and passive port ranges, a one-way password hash, trusted-client CIDRs, total/per-client connection limits, atomic upload replacement, and the same incoming filename templates. FTP and SFTP/SCP reject inbound files beyond the datastore’s 1 GiB per-file ceiling. FTP does not encrypt authentication or file contents; prefer SFTP/SCP wherever possible.

Incoming uploads may keep {filename} or use a pattern such as {timestamp}-{client_ip}-{filename}. The {stem} and {suffix} tokens are also available. Custom naming stores files directly in the selected root; overwrite policy applies to the resulting stored name.

TFTP provides no authentication or encryption. Keep the trusted-client list narrow. Port 1069 works without elevation; binding standard UDP port 69 may require root or CAP_NET_BIND_SERVICE on Linux. The service supports binary RRQ/WRQ transfers plus block-size, timeout, and transfer-size negotiation.

Uploads are committed atomically after the final block. Existing files are rejected unless the administrator explicitly enables overwrite. Up to 20 transfers run concurrently. The latest 1,000 successful and failed transfers are visible in the datastore and can be cleared by an administrator.

Operational limits, diagnostics, and audit history

Administration → Settings groups system controls before account controls. Operational limits bound concurrent and queued automation runs, skip overlapping checks, cap datastore and automation-artifact usage, and preserve a configured amount of free disk.

Administration → System Diagnostics reports web, supervisor, scheduler, and transfer-listener health; scheduler heartbeat age; disk and quota use; SQLite quick-check results; optional command dependencies; numbered migrations; reclaimable orphan artifacts; and the newest administrative audit events.

The audit ledger records the administrator, source address, endpoint, HTTP method, path, result code, and time. Request bodies, passwords, tokens, and other secret values are deliberately excluded.

The supervisor restarts a failed scheduler or enabled transfer worker. Numbered toolkit migrations take a consistent SQLite snapshot before applying a new version.

Backups and future services

Datastore contents are operational files and are intentionally excluded from profile backup/restore. Back up instance/datastore/ separately when those files matter.

TFTP and SSH-transfer settings/history are local operational state. Runtime-only staging is deliberately removed whenever its corresponding service stops.

Background work

Conditions, actions, and automations

Back to contents
The reusable three-part model
  1. A condition observes something or supplies a manual/calendar trigger.
  2. An action defines a trusted response.
  3. An automation connects one condition to actions and adds policy.

Definitions may be shared by many automations. Editing a shared definition pauses every dependent automation for review.

Monitoring policy and states
Check every
Evaluation interval; minimum one second.
Trigger after
Consecutive met checks before actions fire once.
Recover after
Consecutive clear checks before rearming.
Cooldown
Minimum time between incident triggers.

Monitoring moves through healthy, suspect, triggered, recovering, and healthy. Evaluation failures use error and do not count as a met condition. Manual/calendar triggers bypass debounce and recovery.

Parallel actions and sequential stages

Every automation begins with one stage. Actions inside a stage run in parallel; additional stages run sequentially from top to bottom after the previous stage finishes. Stages may be renamed and reordered, and an action can belong to only one stage in a pipeline.

Each stage controls whether the next stage runs: always after completion, only when no action completely fails, or only when every action succeeds. Existing automations are migrated into one default parallel stage.

Webhook actions in later stages can use bounded, non-secret results from earlier stages through {{ '{{actions.results}}' }}, {{ '{{actions.successful}}' }}, {{ '{{actions.partial}}' }}, and {{ '{{actions.failed}}' }}. Raw SSH command output is retained for download but is not automatically exposed to notifications.

ICMP, DNS, and TCP conditions

Ping treats every target as a check. Test displays reachability, RTT, timeout elapsed time, and friendly name/address.

DNS queries each name through each resolver, supports A/AAAA/CNAME/MX/NS/PTR/TXT, and optionally requires any/all expected answers. Test shows every resolver path and timing.

TCP assigns ports per host:

Home FortiGate = gate.example.com | 8443
Google = google.com | 443
Core Switch = 10.0.0.10 | 22, 443, 8000-8010

Expected open requires a connection. Expected closed requires explicit refusal; timeout does not prove closed.

SNMP OID value conditions

SNMP conditions reuse hosts, credentials, and individual OIDs from collections managed in SNMP Tester. Credentials remain in their existing write-only profile store.

Select target hosts once, then add up to 20 named rules. Text rules support equality and containment with optional case sensitivity; numeric rules support inclusive or exclusive upper/lower thresholds. An unavailable-value rule detects failed or empty polls.

Rules use AND logic on each host. A host matches only when every rule matches on that same host, so high CPU on one switch cannot combine with high memory on another. The final threshold requires one, a selected number, or all hosts to match the complete rule group.

Walk rules match when at least one returned value matches. Test results summarize every host and its rule outcomes.

OID profiles can expose reusable calculated values:

Current Memory KB = 1.3.6.1.4.1.example.1.0
Total Memory KB = 1.3.6.1.4.1.example.2.0
calc: Memory Usage % = percent(Current Memory KB, Total Memory KB)

Supported functions are percent(A, B), remaining_percent(A, B), difference(A, B), and sum(A, B). Inputs must resolve to scalar numeric GET values. Unknown labels, circular dependencies, walked inputs, missing/nonnumeric values, and division by zero are rejected or reported.

Certificate health conditions

Monitor up to 20 TLS endpoints with optional friendly names and custom ports:

Home FortiGate = gate.example.com | 8443
Public Portal = portal.example.com | 443

Connection failures, expired certificates, and not-yet-valid certificates always fail. Configure an expiration warning from 0 to 3650 days, and optionally require hostname/IP SAN matching, validation through the toolkit server’s system trust store, and a correctly ordered chain with no likely missing intermediate.

Disable the optional trust, name, or chain policies when monitoring intentional private/self-signed infrastructure. Test results show the endpoint, common name, expiration horizon, TLS version, timing, and every policy failure reason.

Manual and calendar conditions

Manual triggers run only when an administrator selects Run now. Calendar conditions contain up to 50 one-time, daily, weekday, every-N-weeks, monthly-date, or ordinal-weekday rules with an explicit timezone.

Missed occurrences may run late, run only within a grace period, or skip. Refresh next run time is read-only. Simultaneous rules collapse into one occurrence.

SSH collection action

Runs up to 50 hosts concurrently and retains output. Completion is the return of the original device prompt. Default ceiling is 300 seconds; override one command:

[timeout=600] diag debug report

Overrides are 1–3600 seconds with a one-hour combined budget per host. Timeouts retain partial output and stop later commands only on that host.

SSH file collection action

Fetches up to 200 host/path combinations over SFTP, SCP, or FTP using write-only credentials. Use Friendly Name = host targets and the same token-based filename patterns as Multi-Transfer. SFTP/SCP encrypt traffic; FTP does not.

Collected action run retains successful files beside the run and adds them to Download ZIP without placing them in the datastore. Datastore persists files beneath a selected folder and can create one friendly-name/host subfolder per target. Cross-run filename collisions receive a numeric suffix rather than overwriting.

Retained binary files are stored outside SQLite under instance/automation_artifacts/. Deleting, clearing, or retention-pruning a run deletes its artifact directory too.

Syslog notification action

Sends RFC 5424 to up to 20 UDP/TCP collectors:

Primary SIEM = syslog.example.com | 514
Backup collector = 192.0.2.20 | 5514

Messages support {{ '{{trigger.status}}' }}, {{ '{{trigger.summary}}' }}, {{ '{{trigger.met}}' }}, and {{ '{{timestamp}}' }}. UDP success means the local socket accepted the datagram, not remote acknowledgment.

Webhook / API notification action

Sends one POST, PUT, or PATCH template to up to 10 HTTP/HTTPS endpoints. Endpoints may use Friendly Name = URL. Configure accepted status values/ranges, timeout, TLS verification, body format, and write-only headers.

JSON templates are parsed before substitution, so exact trigger and prior-action variables retain boolean/object/list types. Text and JSON templates also support status, summary, and timestamp. Redirects are reported but not followed; retained response previews are capped at 4 KiB per endpoint.

Authorization/API-key headers are encrypted, never rendered back into the editor, and never copied into retained run output. Leave the field blank to keep saved headers, or select Remove all saved headers while updating.

History, downloads, cleanup, and scheduler

Checks, triggers, action status, and structured retained output live in instance/automations.sqlite3; binary transfer artifacts live in instance/automation_artifacts/. SSH ZIPs contain timestamped host files, SFTP/SCP ZIPs include retained fetched files, and syslog ZIPs contain destination JSON. Delete run removes one result and its artifacts; Clear collected data removes action runs but preserves condition history.

Administration → Settings controls global retention. Check records default to 7 days; collected action runs default to never expire. Use 0 for indefinite retention. The card previews eligible records, supports an immediate policy-based prune, and reports database size and oldest records.

The scheduler prunes at most once daily. Optimize database is a separate manual action that reclaims unused SQLite file space and may briefly pause automation writes.

./twn start launches one scheduler beside the web service. Work continues without a browser. Imported automations remain paused.

Packet Replay

Back to contents
Inputs, rewrites, VLANs, and sending

Transmit raw Ethernet frames from hex or classic Ethernet PCAP. Select the wired interface, review decoded fields/rewrite plan, then send from Preview. Multiple source packets, repeat, interval, and MAC/IP/port rewrites are supported.

VLAN fanout accepts IDs, ranges, and untagged: untagged,10-12,20. VLAN 0 is priority-tagged, not untagged.

Permissions and capture verification

Linux needs root or CAP_NET_RAW. macOS uses Scapy/libpcap/BPF and may require elevated BPF access. Wireless 802.11 injection is unsupported.

sudo tcpdump -i en0 -e -nn -vvv 'vlan or ether proto 0x8100'
sudo tcpdump -i en0 -e -nn -vvv 'vlan and port 514'

Sender acceptance is not remote delivery; verify actual egress on the wire.

Operations and troubleshooting

Service commands
./twn start            Start web service and scheduler
./twn stop             Stop both processes
./twn restart          Restart both processes
./twn status           Show process status
./twn logs             Show recent errors
./twn enable-https     Generate and enable local HTTPS
./twn disable-https    Return to HTTP; retain certificate files
./twn fix-permissions  Repair instance ownership after sudo
./twn adminreset       Reset users only
./twn reset-data       Remove saved toolkit data
HTTPS and self-signed certificates

Fresh installations use HTTPS by default; upgrades preserve their existing protocol. Run ./twn enable-https to migrate an older HTTP installation and safely restart the toolkit. Add stable names or addresses when needed: ./twn enable-https toolkit.local 192.0.2.25.

The certificate includes localhost, loopback, and detected machine names. Browsers warn until the certificate is trusted. HTTPS enables secure session cookies, and ./twn status prints the active HTTPS URL.

The owner-only private key lives at instance/tls/key.pem and is not included in profile backups. Run ./twn disable-https to restart over HTTP without deleting the certificate. Externally managed PEM files may be supplied with both TWN_TOOLKIT_CERTFILE and TWN_TOOLKIT_KEYFILE.

Administration → Settings can define a short instance name and preferred FQDN. Validation checks DNS-name structure but never performs a lookup. The short name appears in browser titles and the sidebar; the FQDN is used in launcher URLs. Explicitly select certificate regeneration while saving when the toolkit-managed certificate should include the new identity.

Port conflicts and sudo ownership

Find a Linux listener with sudo ss -lptn 'sport = :5050' or sudo lsof -nP -iTCP:5050 -sTCP:LISTEN. On macOS use lsof -nP -iTCP:5050 -sTCP:LISTEN.

A sudo launch can create root-owned instance/ files. Run ./twn fix-permissions before returning to normal-user operation.

Common checks
  • Restart after code/dependency updates.
  • Verify both web and scheduler with ./twn status.
  • Inspect ./twn logs.
  • Test saved Fortinet profiles.
  • Check target, interface, firewall, remote permissions, and timeout.
  • Hard-refresh if browser assets look stale.
Responsibility

Security, local data, and disclaimer

Back to contents
Deployment boundary and retained data

This is local-first software for a trusted workstation/internal network, not direct internet exposure. Use TLS and an appropriate access layer across untrusted networks. Server access settings can allow explicit addresses/CIDRs; loopback remains allowed.

User passwords are scrypt hashes; sessions use a separate secret. Profile stores and automation databases live under owner-only ignored instance/ paths. Protect the host, filesystem backups, exported backups, downloads, and browser session.

Use at your own risk

The WiFi Ninja’s Toolkit is provided as-is, without warranty. Some tools change configuration, authenticate, scan, query logs, transmit crafted packets, or execute commands with saved credentials.

You are responsible for authorization, protecting credentials/output, reviewing previews, selecting correct targets/interfaces, and understanding impact. Use at your own risk.
